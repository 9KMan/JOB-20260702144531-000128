# Telegram 409 Conflict — Root cause and fix

**Date:** 2026-06-30
**Symptom:** Bot @SuperTuiNuiBot stopped responding to plain chat messages ("hi" etc.). Replies to inbound messages took 170s+ or never arrived.
**Affected:** All Hermes gateway users on `vmi3077922.contaboserver.net` (this VPS).
**Bot token:** stored in `/home/deploy/.hermes/.env` as `TELEGRAM_BOT_TOKEN`.

## Root cause

A second `getUpdates` consumer was running on **this same box**, competing with the
Hermes gateway's long-polling PTB instance. Telegram's Bot API allows exactly one
`getUpdates` consumer per bot token — the loser gets HTTP 409 Conflict on every call.

The duplicate poller was a cron job:

```
*/1 * * * * python3 /home/deploy/.hermes/autonomous-factory/telegram_poll_commands.py >> /home/deploy/.hermes/logs/telegram-poll.log 2>&1
```

It was supposed to parse CEO approval commands (`a`/`A`/`c`/`s` against
`JOB-XXXXXXXX-YYYYYY` IDs) and write JSON events to
`~/.hermes/jobs/{job_id}/events/`. Its docstring claimed
*"Uses offset-based getUpdates — no 409 conflict with gateway PTB"*.
**That claim is wrong**: `offset` controls which *messages* you receive, not whether
you can call `getUpdates` concurrently. The Bot API rejects any second concurrent
poller regardless of offset.

The cron started every minute, blocked ~5s on `getUpdates?timeout=5`, and got a
409 every time the Hermes gateway was actively long-polling (which is ~95% of the
time). Meanwhile the gateway kept retrying every 20s and getting 409'd back,
producing the cascade visible in `gateway.log` from 00:33 onwards.

## Fix applied

Disabled the cron entry. Original crontab backed up at:

```
/home/deploy/.hermes/autonomous-factory/crontab.backup.20260630
```

The disabled entry is preserved as a comment in the live crontab so the operator
can see what was changed. Recovery is a single uncomment if needed:

```bash
# Restore from backup
crontab /home/deploy/.hermes/autonomous-factory/crontab.backup.20260630

# Or just uncomment the line in current crontab -l output.
```

**Do not re-enable this cron** unless you have moved the script off `getUpdates`.
The Telegram Bot API is fundamentally single-consumer per bot token; there is no
`offset` trick that lets two pollers coexist. If you re-enable it, you will
recreate the same 409 storm within minutes.

## Verification

- Before fix: gateway.log shows ~6 conflicts per 5 minutes.
- After fix: zero conflicts since 01:02:43 local time.
- End-to-end: "hi" messages get replies within seconds (was 174s before).

## Open follow-ups (not blocking chat, but real)

### 1. CEO a/A/c/s command routing is currently broken

The disabled script was the only thing parsing CEO approval commands
(`a JOB-...`, `s JOB-...`, `c JOB-...`) and writing events to
`~/.hermes/jobs/{job_id}/events/`. Without it, the approval pipeline no longer
receives CEO decisions through the bot.

**Three options to restore:**

a. **Gateway-side handler (proper fix).** Add a slash-command or message-prefix
   handler to the Hermes Telegram adapter that runs the same parsing logic
   (`parse_command` in `telegram_poll_commands.py:111-136`) and writes the same
   event files. Place: `plugins/platforms/telegram/adapter.py` near the existing
   `_handle_command` / `_handle_text_message` dispatch.

b. **Use the gateway's built-in `/approve` slash command** instead of the
   bespoke a/A/c/s format. The gateway already supports approval workflows
   (see `_handle_callback_query` in adapter.py — handles inline-button
   approvals). Migrate CEO to clicking buttons rather than typing commands.

c. **Accept the regression.** Document that CEO approvals now require the
   approval UI instead of chat commands. The pipeline's other
   approval/skip/confirm paths may still work depending on how it triggers them.

### 2. Investigate why `telegram-poll.log` was always empty

The cron entry redirected stdout/stderr to `telegram-poll.log`, but the file had
zero lines at the time of diagnosis. Either cron was failing silently to invoke
the script (environment issue, missing python3 in cron PATH, etc.) or the script
was crashing before any logging happened. If we ever need to re-run the script,
fix this first — silent cron failures are the worst kind of bug.

### 3. The script's docstring is misleading

It still claims "no 409 conflict". If option 1a is chosen, the parsing logic
will move out of this file entirely and the docstring becomes irrelevant. If
option 1b or 1c, at minimum update the docstring to reflect the real situation
so future engineers don't fall for it.
